# DATA PROCESSING BRIEF — PMM MARKET INTELLIGENCE AGENT

---

**Prepared for:** Data Protection Officer
**Project:** PMM Market Intelligence Agent
**Date:** May 2026
**Status:** MVP / Lab prototype (single authenticated user; testing mode only)

---

### What is this system?

The PMM Market Intelligence Agent is an autonomous workflow that takes a company URL as input and delivers three structured PMM outputs without human intervention. A user submits a URL through a Gradio interface. The agent researches the target company via web search, pulls the last 10 messages from the authenticated user's Gmail inbox as proxy CRM context, and reads documents from the user's Google Drive. All three data streams are merged and fed into three output nodes running in parallel: a Market Research Deck and an ICP One Pager saved automatically to Google Drive, and a Sales Email written and sent autonomously via Gmail. No human reviews any output before it is saved or sent.

---

### 1. What personal data does the system process?

The system processes four categories of personal data:

**Gmail inbox content (third-party communications data):**
- Subjects, snippets, sender names, and email addresses from the last 10 messages in the authenticated user's inbox
- Message body content, which includes communications authored by third parties who are not party to this system and have no knowledge their data is being processed

**Google Drive document content (may contain research participant data):**
- Full text of Google Docs files stored in the user's Drive, which may include UXR interview transcripts, research notes, and documents containing names, roles, organisations, and verbatim quotes of research participants

**Email recipient identity (inferred by LLM):**
- The name and email address of the sales email recipient, autonomously inferred by GPT-4o-mini from Gmail and Drive content
- This individual receives an outbound email without any prior knowledge of or opportunity to object to the processing

**User account identity:**
- The authenticated Google account email address (olalla.murciego@gmail.com) used for all OAuth2 operations

**No special-category data is intentionally collected.** However, the agent has no mechanism to detect or exclude Article 9 data from Google Drive documents before passing them to OpenAI. If UXR documents contain health, ethnicity, or other sensitive data from research sessions, that data would be processed without an Article 9 condition in place.

### 2. Where does the data come from?

| Data | Source |
|---|---|
| Gmail inbox content (subjects, snippets, sender info, message bodies) | Pulled directly from the user's Gmail inbox via Gmail API (OAuth2) on each request |
| Google Drive document content | Pulled directly from the user's Google Drive via Drive API v3 (OAuth2) on each request; up to 5 documents per session |
| Target company research data | Retrieved by OpenAI's built-in web search tool from public web sources |
| Email recipient identity | Inferred autonomously by GPT-4o-mini from Gmail and Drive content — not supplied by the user |
| User account identity | Captured via Google OAuth2 authentication flow |

---

### 3. What is the data used for?

| Purpose | Description |
|---|---|
| **P1 — Market Research Deck generation** | Gmail content and Drive documents are merged with web research results and sent to GPT-4o-mini to generate a structured company overview, competitive landscape, and GTM recommendations; saved as a Google Doc |
| **P2 — ICP One Pager generation** | Drive document content and web research results are sent to GPT-4o-mini to generate an Ideal Customer Profile; saved as a Google Doc |
| **P3 — Sales Email generation and autonomous send** | Gmail content is used as CRM context to personalise an outbound sales email generated by GPT-4o-mini; the email is sent autonomously via Gmail Send to a recipient inferred by the LLM — no human review before delivery |
| **P4 — LLM inference across all nodes** | All input data (Gmail content, Drive documents, web search results) is sent to OpenAI's API (GPT-4o-mini) for inference; this constitutes an EEA-to-US international transfer on every session |

**Key finding for DPO:** Gmail content is used for a purpose entirely different from the one for which it was originally collected (personal email communication). This is a textbook purpose limitation failure under Art. 5(1)(b). The original collection purpose and the agent's use are incompatible, and no exception under Art. 6(4) has been assessed.

---

### 4. Who processes the data?

| Entity | Role | What they receive |
|---|---|---|
| The agent system (n8n workflow, running on n8n Cloud or self-hosted) | Internal orchestration layer | Routes all data between nodes; no persistent storage of personal data by default |
| **OpenAI API (USA)** | **External processor — third country** | Receives on every session: Gmail message content, Google Drive document text, web search results, and inferred recipient identity; processes all data via GPT-4o-mini for inference and via the embeddings API for any vector operations |
| **Google (Gmail, Drive, Docs APIs)** | Processor (API operations) / Controller (account data) | Reads Gmail inbox; reads and writes Google Drive; sends email via Gmail Send; stores output Google Docs; retains message history per Google's own retention settings |
| **Gradio / Jupyter** | Local interface layer | Receives the URL input from the user and passes it to the n8n webhook; local processing only in current configuration |

---

### 5. Where is data stored and processed?

| Data store | Location | Notes |
|---|---|---|
| OpenAI API | USA | Gmail content and Drive document text sent in API requests on every session; OpenAI's data retention policy for API inputs applies but has not been documented for this project |
| Google Drive (output files) | EU or USA depending on Google account settings | Market Research Deck and ICP One Pager saved as Google Docs; no automated deletion schedule in place |
| Gmail (sent email) | USA (Google infrastructure) | Autonomously sent sales email retained in Gmail Sent folder; no deletion schedule |
| n8n | EU (n8n Cloud) or depends on host (self-hosted) | Passes data in transit; no persistent storage of personal data by default |
| Gradio / Jupyter | Local machine | Interface only; no data persisted locally |

**Key finding for DPO:** All three primary external processors (OpenAI, Google Gmail, Google Drive) are US-based entities. Every session involving Gmail or Drive data constitutes an EEA-to-US international transfer. No Chapter V transfer mechanism has been formally documented for any of these relationships. No Data Processing Agreement has been executed with OpenAI for this project. No DPA exists between the controller and Google for personal Gmail accounts — Google Workspace carries a DPA; personal accounts do not.

---

### 6. Does the system make or assist in decisions that affect people?

Yes — and this is the most significant compliance risk in the system.

The agent autonomously selects a sales email recipient by inferring their identity from Gmail inbox content and Google Drive documents, then sends an outbound email to that person without any human review before delivery. The recipient has no prior knowledge of the processing, no opportunity to object, and no mechanism to contest the action before it occurs.

This constitutes automated processing with real-world consequences for an individual who is not a user of the system and has not consented to any aspect of it.

**Art. 22 GDPR assessment:** While the autonomous email send may not constitute a decision with strictly legal effects, it produces a similarly significant effect on the recipient — unsolicited commercial contact based on LLM-generated profiling of their identity and relevance. The safeguards required under Art. 22(2) and (3) — human intervention mechanism, right to contest, and explanation of the decision logic — are not implemented. This processing activity should be suspended or redesigned to include mandatory human review before any email is sent.
